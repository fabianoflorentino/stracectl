
# v1.0.112 — 2026-03-29 — Privacy improvements: redaction, capture controls, and secure outputs

Summary

- Release focused on privacy: a new event-processing pipeline, redaction enabled by default, and fine-grained capture controls.

Key changes

- Full implementation of the "filter → redact → format → output" architecture to ensure sensitive data is masked before any formatting or output.
- Default `redactor` enabled; applies built-in redaction patterns (JWTs, emails, IPs, common token/secret parameters).
- New modular packages: `tracer`, `filters`, `redactor`, `formatter`, and `output`.

New CLI flags

- `--privacy-level=low|medium|high` — default `high` (minimizes data exposure).
- `--full` — enable full payload capture; requires explicit confirmation or `--force` (warning displayed).
- `--no-args` — do not capture syscall arguments.
- `--max-arg-size=N` — limit argument sizes (default 64).
- `--redact` / `--redact-patterns` — control redaction and add extra patterns.
- `--syscalls` / `--exclude` — syscall include/exclude lists.
- `--ephemeral` / `--ttl` — ephemeral output with automatic expiration.
- `--explain` — dry-run mode that prints what will be captured and which redaction patterns are active.

Behavior & security

- Redaction is applied before any formatting; formatters receive already-masked events.
- File outputs are created with secure permissions (mode 0600) when applicable.
- Ephemeral file mechanism with TTL and a cleanup routine to avoid unintended retention.
- Optional audit logging: metadata and hashes (no raw sensitive content) for enterprise use.

Included redaction patterns

- JWTs, emails, IPv4 addresses, common token/secret/password parameters, and deterministic truncation of long arguments.

Tests & quality

- New unit tests for `filters`, `redactor`, and `formatter`.
- End-to-end integration tests for the pipeline (validating redaction prior to output).
- Fuzzer target for the `redactor` to ensure robustness against arbitrary input.

Documentation

- Updated privacy documentation and safe-usage guides (`docs/privacy-implementation-plan.md`, `docs/privacy-improvements.md`).
- `--explain` implemented to safely inspect the capture surface.

Migration notes / Compatibility

- Default behavior changed to `--privacy-level=high` (more restrictive). Users requiring full capture must opt in with `--full` and accept the explicit warning.
- Review scripts/integrations that expect syscall arguments, as `--no-args` / `--max-arg-size` may reduce available information.

Credits

- Privacy architecture and redaction pattern design: privacy team and community contributors.

Quick test

- Run `stracectl explain` to preview what will be captured without exposing data.
- For full capture (use with caution): `stracectl trace --full --force <args>`
